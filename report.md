# Отчет по разработке MVP Travel AI Агента

## 1. Выбор технического стека

При разработке системы был выбран следующий стек:

- **LangGraph**: Выбран в качестве основного фреймворка для управления состоянием (State Management) и построения сложных циклических графов. В отличие от линейных цепочек, LangGraph позволяет реализовать итеративный процесс фактчекинга, где агент может возвращаться к поиску, если текущей информации недостаточно.
- **OpenRouter (Gemini Flash)**: Использование OpenRouter обеспечивает гибкость в выборе моделей. Модели Gemini (в частности `flash-preview`) были выбраны из-за большого контекстного окна, что критично при анализе множества вырезок с веб-сайтов, и высокой скорости работы.
- **DuckDuckGo Search (DDGS)**: Выбран как основной поисковой движок, так как он бесплатен, не требует API-ключей для базового использования и предоставляет стабильное API для получения текстовых сниппетов.
- **BeautifulSoup4 & Requests**: Классический стек для извлечения контента. Для обхода блокировок интегрирован прокси-сервис `r.jina.ai`.

## 2. Архитектурные решения

### Многоагентная система (3 субагента)

Для обеспечения полноты ответа задача была разделена между тремя специализированными субагентами. Так сделали так как есть важные вопросы в вопросах путешествий которые покрыть одним агентом достаточно трудно. 

1. **Official Docs Agent**: Фокусируется на визовых требованиях, официальных правилах въезда и документах.
2. **Reviews Agent**: Анализирует живой опыт путешественников (Reddit, форумы), выявляя реальные очереди, особенности работы пограничников и подводные камни.
3. **Practical Recs Agent**: Собирает информацию по бытовым вопросам (адаптеры, SIM-карты, повербанки, таможенные ограничения на технику).

### Двухэтапная верификация (Фактчекер)

Точность в нашей тематике критична поэтому

- Внедрен отдельный **Fact-checker**, который работает в цикле LangGraph. 
- Он получает финальный агрегированный ответ и проверяет каждое утверждение через независимый поиск.
- Только после получения статуса `APPROVED` от фактчекера ответ отдается пользователю.

## 3. Сложности и их решение

В ходе разработки мы столкнулись с рядом проблем, которые были успешно решены:


| Проблема                                                          | Решение                                                                                                                                                                                                                                                |
| ----------------------------------------------------------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------ |
| **Блокировки парсинга** (особенно TripAdvisor и крупные порталы). | Использование `-tripadvisor` в поисковых запросах который убирает его из выдачи и внедрение fallback-механизма через `r.jina.ai`. При этом часть проблем решается естественным образом тк если иформации не хватает - агенты продолжают ходить в поиск |
| **Дублирование поисковых запросов**.                              | Внедрена дедупликация на уровне формирования `seed_queries` и передача истории поиска (`queries_used`) агенту, чтобы он видел, что уже искал.                                                                                                          |
| **Бесконечные циклы / Глубина рекурсии агента**                   | В LangGraph и субагентах установлены жесткие лимиты: `max_rounds` и `max_pages`. Если агент не нашел инфу за какое-то количество раундов, он обязан сделать выводы из того, что есть.                                                                  |
| **"Галлюцинации" фактчекера**.                                    | Иногда сам фактчекер начинал придираться к корректным данным или зацикливался на переписывании. Решено ограничением количества итераций переписывания (`rewrite_count < 2`) и строгим промптом на использование найденных источников.                  |
| **Избыточность контекста**.                                       | При парсинге сайтов LLM может "захлебнуться" в HTML-мусоре. Решено через `extract_visible_text` (удаление скриптов, стилей, навигации) и жесткую обрезку текста (`truncate_text`) по 12к символов.                                                     |
